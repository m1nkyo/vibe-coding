01 — Definition

바이브 코딩이란 무엇인가

바이브 코딩은
대형 언어 모델(LLM)에 프롬프트를 입력해
문제를 풀어내는 AI 기반 프로그래밍 방식이다.

프로그래머는 더 이상 코드를 직접 타이핑하지 않는다.
대신 AI와 프롬포트로 소통하면서
ai가 생성한 코드를 테스트하고 수정하며,
방향을 잡아주는 역할로 옮겨간다.

shuffle.js — AI generated
// "카드 한 벌을 섞는 함수를 작성해줘"
function shuffle(deck) {
  for (let i = deck.length - 1; i > 0; i--) {
    const j = Math.floor(Math.random() * (i + 1));
    [deck[i], deck[j]] = [deck[j], deck[i]];
  }
  return deck;
}
02 — Origin

용어의 탄생

Feb 2025

바이브코딩는
2025년 2월 안드레이 카파시(Andrej Karpathy)가 처음 꺼냈다.

이는 LLM에 의존하는 코딩 접근 방식을 말하며
프로그래머가 코드를 직접 작성하는 대신
자연어를 통해 코드를 생성하는 것이다.

"실제로 코딩은 아니다.
저는 그저 사물을 보고, 말하고, 실행하고,
복사해서 붙여넣기만 하면
대부분 작동한다. "
— Andrej Karpathy
03 — Core Idea

"가장 인기 있는 새 프로그래밍 언어는 영어다"


바이브 코딩은 카파시가
2023년"가장 인기 있는 새로운 프로그래밍 언어는 영어"라고 했던 주장에서 한발 더 나아간 것이다.

LLM이 충분히 똑똑해진 지금,
인간은 컴퓨터에 명령하기 위해
더 이상 특정 프로그래밍 언어를 배울 필요가 없다는 뜻이다.

BEFORE

구문을 배운다

문법과 자료형, API를 익혀서
한 줄 한 줄 정확하게 타이핑한다.

VIBE

의도를 말한다

원하는 결과를 자연어로 설명하면 AI가 코드를 만들어낸다.

AFTER

방향을 잡는다

일단 실행해보고, 테스트하고,
다시 프롬프트로 고칠 방향을 일러준다.

04 — The Boundary

어디까지가 바이브 코딩인가


핵심은 사용자가 코드를 온전히 이해하지 못한 채
코드를 수용한다는 것에 있다.

만약 모든 줄을 일일이 검토하고 테스트하고 이해했다면, 그건 바이브 코딩이 아니다.

AI 연구자 사이먼 윌리슨(Simon Willison)은
그 경계를 분명히 그었다.

"모든 걸 검토하고 테스트하고 이해했다면,
그건 바이브 코딩이 아니다.
LLM을 그냥 타이핑 보조 도구로 쓴 것일 뿐이다."
— Simon Willison
05 — In Practice

"한 사람을 위한 소프트웨어"


비개발자인 뉴욕타임스 기자 케빈 루즈(Kevin Roose)는
바이브 코딩으로 작은 앱 여러 개를 직접 만들어봤다.
그는 이걸 개인의 필요에 딱 맞춘 '한 사람을 위한 소프트웨어'라고 불렀다.

예전 같으면 엔지니어링 팀이 통째로 붙어야 했을 소프트웨어를,
이제는 개인 혼자서도 만들 수 있게 된 셈이다.

CASE · LunchBox Buddy

🥗 점심 도시락 추천 앱

냉장고 속 재료를 분석해서
도시락에 넣을 만한 음식을 추천해주는 개인 맞춤형 도구다.
프로그래머가 아니어도 제대로 작동하는 소프트웨어를 만들 수 있다는 걸
보여준 사례다.

08 — Trade-offs

가능성과 한계

+ 가능성

  • 훈련·기술 없이도 코드를 생산
  • 비개발자도 기능하는 앱 제작
  • 개인 맞춤형 도구를 빠르게 구현
  • 프로토타입 제작에 탁월
  • 팀 없이 개인 수준 개발 가능
VS

− 한계

  • 결과가 제한적이고 오류 발생 잦음
  • AI가 가짜 리뷰를 조작한 사례도
  • 코드를 이해 못한 채 사용
  • 감지되지 않은 버그의 위험
  • 취미 프로젝트에 더 적합
09 — Critique

독창성인가, 재생산인가

"루즈의 열광은 독창성이 아니라
재생산에서 나온 것이다."
— Gary Marcus, AI 전문가

AI 전문가 게리 마커스(Gary Marcus)는,
루즈의 런치박스 버디 앱을 만든 알고리즘이
애초에 비슷한 일을 하던 기존 코드로 학습됐을 거라고 꼬집었다.

새로 창조해낸 게 아니라,
학습한 패턴을 다시 짜맞춘 것에 가깝다는 비판이다.

10 — Responsibility

이해와 책임의 문제


바이브 코딩은 이해와 책임이라는 문제를 함께 불러온다.

개발자가 AI가 짠 코드의 동작을
제대로 이해하지 못한 채 사용하면,
미처 발견하지 못한 버그와 오류가 그대로 남을 수 있다.

✓ 적합

프로토타입 제작

아이디어를 빠르게 검증하고 형태부터 잡아보는 단계에서는
확실히 강력하다.

⚠ 위험

전문적 프로덕션 환경

기존 시스템을 고쳐 나가고 디버깅과 유지보수를 이어가야 하는 곳에서는
코드의 품질과 이해도가 무엇보다 중요하다.

11 — The Irony

AI가 코딩을 거부할 때

아스 테크니카는 윌리슨의 말을 빌려,
이를 "바이브 코딩의 부상에서 나타난 아이러니한 반전"이라고 표현했다.

개발자가 코드를 짜달라고 했더니,
되레 AI가 거부하면서 로직은 직접 만들라고

12 — Deployment

코드를 넘어 배포까지


바이브 코딩으로 만든 코드를 웹에 올리는 과정에는
여전히 서버 설정이나 도메인 관리 같은 기술 지식이
필요할 때가 많다.

이 문턱을 낮춰주는 도구들도 속속 나오고 있다.

HOSTING

Vercel · Netlify

기존 호스팅 서비스를 이용해 정적·동적 사이트를 배포한다.

CONVERSATIONAL

대화형 배포 플랫폼

AI 도구와 곧바로 연동해,
대화하듯 배포까지 처리하는 서비스가 떠오르고 있다.

MCP-BASED

AI 스페이스

모델 컨텍스트 프로토콜(MCP)을 기반으로
코드 생성부터 배포까지 한 번에 처리한다.

13 — Takeaways

핵심 정리

  • 01새로운 방식이다. 코드를 직접 타이핑하는 대신, 자연어로 의도를 전하면 AI가 코드를 만들어낸다.
  • 02경계는 '이해'에 있다. 코드를 일일이 검토하고 이해했다면, 그건 바이브 코딩이 아니라 그냥 타이핑 보조다.
  • 03문턱이 낮아졌다. 비개발자도 '한 사람을 위한 소프트웨어'를 직접 만들 수 있게 됐다.
  • 04한계도 분명하다. 오류와 버그의 위험, 재생산이라는 비판이 따라붙는다. 프로토타입엔 잘 맞지만, 프로덕션에선 신중해야 한다.
